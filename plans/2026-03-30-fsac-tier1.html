FSAC Tier 1 Improvements Implementation Plan

For Claude: REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

Goal: Implement four Tier 1 safety and observability improvements from docs/plans/fsac-learnings.md.

Architecture: All changes are self-contained within existing modules. No new types or public APIs. Semaphore is module-level state in AstAnalyzer.fs. Logging goes to eprintfn (stderr). Build runner gets async stdout/stderr reads + timeout.

Tech Stack: F# on .NET 10, FSharp.Compiler.Service, xUnit v3, Swensen.Unquote (test <@ ... @>)

Task 1: Script checking semaphore

Files: - Modify: src/TestPrune.Core/AstAnalyzer.fs (around line 729 — getScriptOptions) - Test: tests/TestPrune.Tests/AstAnalyzerTests.fs

Context: GetProjectOptionsFromScript has internal FCS state corruption under concurrent calls. We serialize with a SemaphoreSlim(1, 1). The semaphore is held only during the GetProjectOptionsFromScript call, not the rest of getScriptOptions.

Step 1: Write a failing test

Add this test to AstAnalyzerTests.fs at the end of the file:

module ``getScriptOptions concurrency`` =

    [<Fact>]
    let ``concurrent calls do not corrupt each other`` () =
        let checker = FSharpChecker.Create()
        // Run two concurrent getScriptOptions calls and verify both return valid options
        let source1 = "module A\nlet x = 1"
        let source2 = "module B\nlet y = 2"
        let f1 = async { return! getScriptOptions checker "/tmp/a.fsx" source1 }
        let f2 = async { return! getScriptOptions checker "/tmp/b.fsx" source2 }
        let results = [ f1; f2 ] |> Async.Parallel |> Async.RunSynchronously
        // Both should return options with at least the requested file in SourceFiles
        test <@ results[0].SourceFiles |> Array.exists (fun f -> f.EndsWith("a.fsx")) @>
        test <@ results[1].SourceFiles |> Array.exists (fun f -> f.EndsWith("b.fsx")) @>

Step 2: Run test to verify it fails or passes without the semaphore

dotnet test tests/TestPrune.Tests --filter "getScriptOptions concurrency" -v minimal

Note: This test may pass already (FCS doesn't always corrupt under concurrency). Its value is as a regression guard.

Step 3: Add the semaphore to AstAnalyzer.fs

Near the top of AstAnalyzer.fs, after the existing #nowarn "57" line, add:

open System.Threading

/// Serializes concurrent GetProjectOptionsFromScript calls.
/// FCS has internal state corruption when script options are loaded concurrently.
let private scriptSemaphore = new SemaphoreSlim(1, 1)

Then in getScriptOptions (currently around line 730), wrap only the GetProjectOptionsFromScript call:

let getScriptOptions (checker: FSharpChecker) (sourceFileName: string) (source: string) =
    async {
        let sourceText = SourceText.ofString source

        do! scriptSemaphore.WaitAsync() |> Async.AwaitTask

        let! projOptions, _diagnostics =
            try
                checker.GetProjectOptionsFromScript(sourceFileName, sourceText, assumeDotNetFramework = false)
            finally
                scriptSemaphore.Release() |> ignore

        // ... rest of the function unchanged

Expected: PASS

Step 3: Add the helpers to AstAnalyzer.fs

Add these functions before getScriptOptions:

/// Resolve a path to absolute using basePath as the base directory.
let internal resolveToAbsolute (basePath: string) (path: string) =
    if Path.IsPathRooted(path) then path
    else Path.GetFullPath(Path.Combine(basePath, path))

/// Resolve relative -r: reference paths in FCS OtherOptions to absolute paths.
let private resolveReferenceOptions (baseDir: string) (opts: string array) =
    opts
    |> Array.map (fun opt ->
        if opt.StartsWith("-r:", StringComparison.Ordinal) then
            let path = opt[3..]
            "-r:" + resolveToAbsolute baseDir path
        else
            opt)

Step 4: Apply in getScriptOptions

        let baseDir = Path.GetDirectoryName(sourceFileName)

                OtherOptions =
                    projOptions.OtherOptions
                    |> resolveReferenceOptions baseDir }

        return enhancedOptions

Task 3: Build timeout + duration logging

Files: - Modify: src/TestPrune/Program.fs (in dotnetBuildRunner, around line 93)

Context: The current dotnetBuildRunner reads stdout then stderr sequentially (can deadlock if stderr buffer fills while waiting for stdout). We fix that with async reads and add a 10-minute timeout. On timeout, kill the process and return exit code 1.

There is no good unit test for this without spawning a real hanging process. Verify manually by running mise run example.

Step 1: Add the timeout constant and rewrite dotnetBuildRunner

Replace the existing dotnetBuildRunner function in Program.fs:

let private buildTimeoutMs = 600_000 // 10 minutes

/// Default build runner: runs `dotnet build` on the solution with a 10-minute timeout.
let dotnetBuildRunner: BuildRunner =
    fun (repoRoot: string) ->
        let slnFiles =
            [| "*.slnx"; "*.sln" |]
            |> Array.collect (fun pattern -> Directory.GetFiles(repoRoot, pattern))

        let slnPath = if slnFiles.Length > 0 then slnFiles.[0] else repoRoot

        let buildPsi = ProcessStartInfo("dotnet", $"build \"%s{slnPath}\" -v quiet")
        buildPsi.UseShellExecute <- false
        buildPsi.RedirectStandardOutput <- true
        buildPsi.RedirectStandardError <- true

        use buildProc = Process.Start(buildPsi)
        let sw = Diagnostics.Stopwatch.StartNew()

        // Read stdout and stderr asynchronously to avoid deadlock if a buffer fills.
        let stdoutTask = buildProc.StandardOutput.ReadToEndAsync()
        let stderrTask = buildProc.StandardError.ReadToEndAsync()

        let completed = buildProc.WaitForExit(buildTimeoutMs)

        if not completed then
            buildProc.Kill(entireProcessTree = true)
            eprintfn $"Build timed out after {buildTimeoutMs / 60_000} minutes — aborting index"
            1
        else
            // Ensure async reads complete before returning.
            stdoutTask.Wait()
            stderrTask.Wait()
            sw.Stop()
            eprintfn $"[dotnet build] → exit %d{buildProc.ExitCode} in %.1f{sw.Elapsed.TotalSeconds}s"
            buildProc.ExitCode

Task 4: Process execution logging in TestRunner

Files: - Modify: src/TestPrune.Core/TestRunner.fs (in runProcess, around line 22) - Test: tests/TestPrune.Tests/TestRunnerTests.fs

Context: Add duration logging to runProcess so test invocations show timing on stderr. The runAllTestsWith and runFilteredTestsWith functions use an injectable runner, so logging only affects the real runProcess used in production. Existing tests using fakeRunner are unaffected.

Step 1: Modify runProcess in TestRunner.fs

Replace the existing runProcess:

let private runProcess (fileName: string) (arguments: string) : TestResult =
    let psi = ProcessStartInfo(fileName, arguments)
    psi.RedirectStandardOutput <- true
    psi.RedirectStandardError <- true
    psi.UseShellExecute <- false
    psi.CreateNoWindow <- true

    use proc = Process.Start(psi)
    let sw = Diagnostics.Stopwatch.StartNew()

    // Read async to avoid deadlock if a buffer fills while waiting for the other.
    let stdoutTask = proc.StandardOutput.ReadToEndAsync()
    let stderrTask = proc.StandardError.ReadToEndAsync()
    proc.WaitForExit()

    let stdout = stdoutTask.Result
    let stderr = stderrTask.Result
    sw.Stop()

    eprintfn $"[%s{fileName} %s{arguments}] → exit %d{proc.ExitCode} in %.1f{sw.Elapsed.TotalSeconds}s"

    { ExitCode = proc.ExitCode
      Output = stdout + stderr }

Note: open System.Diagnostics is already present via ProcessStartInfo. Add open System.Threading.Tasks if needed, but Task is already used via ReadToEndAsync() which returns Task<string> — .Result is fine here since we already called WaitForExit().
